## 3. 폴더 구조 및 전체 흐름

```
team-project/
├── README.md                ← 보고서 (본 문서)
├── main.py                  ← 전체 프로그램 통합 실행 진입점
├── 1/                       ← 팀원 1의 개별 모듈
│   └── tool.py
├── 2/
│   └── tool.py
├── 3/
│   └── tool.py
├── 4/
│   └── tool.py
├── data/                    ← 각 모듈에서 공유 혹은 개별로 사용하는 데이터
│   └── ...
└── requirements.txt
```

- `main.py`: 선택 메뉴 기반 실행
- `1/tool.py` 등 각 디렉토리는 독립적인 `run()` 함수 포함
- `data/`: 모듈 간 간섭 없이 JSON, CSV 등 파일 분리 저장

## 5. Git 협업 전략 및 구조

### 5.1 브랜치 전략

- 모든 팀원은 `main` 브랜치에 직접 작업하지 않고, 개인 기능은 다음과 같은 브랜치명으로 관리함

```
feature/1-tool
feature/2-tool
feature/3-tool
feature/4-tool
```

- 통합 전에는 `develop` 브랜치에서 기능 병합을 먼저 진행하고, 최종적으로 `main`에 통합

---

### 5.2 Commit 메시지 전략

- 규칙:
  - `feat`: 새로운 기능 추가
  - `fix`: 버그 수정
  - `docs`: 문서 작성
  - `refactor`: 리팩토링
  - `test`: 테스트 코드
- 예시:
  ```
  feat: add income input function to budget tool
  fix: resolve index out of range in quiz
  ```

---

### 5.3 Pull Request & 병합 방식

- 각 브랜치에서 기능 개발 후 PR(Pull Request)을 생성
- PR 설명에는 기능 설명, 주요 변경 코드 캡처, 테스트 방법 기입
- 충돌 발생 시 `develop` 브랜치에서 테스트 후 `main`으로 merge

---

### 5.4 GitHub Issues 및 협업 툴 사용

- GitHub Issues에 작업 항목 등록 및 할당
- 프로젝트 보드로 업무 흐름 관리
- 예시 이슈:
  - `[기능] 단어장에 단어 검색 기능 추가`
  - `[버그] 날짜 형식 오류 발생`

## 7. 개발 중 문제와 해결 과정

| 발생 문제 | 원인 분석 | 해결 방법 |
|-----------|-----------|-----------|
| 병합 충돌 발생 | 파일 위치 충돌 | 각 기능을 개별 폴더로 분리하여 병합 충돌 방지 |
| 데이터 덮어쓰기 | 각 기능이 동일한 data.json 사용 | 모듈별 별도 데이터 파일 분리 |
| 브랜치 혼동 | 직접 main에 커밋 시도 | README에 브랜치 전략 명시 + 팀장 리뷰 |

## 8. 프로젝트 회고 및 결론

### 8.1 Git 협업을 통해 얻은 점

- 브랜치 전략과 PR 리뷰를 통해 버전 관리를 실제처럼 경험함
- merge conflict 해결 및 commit rollback 등의 실습 경험이 유익했음
- 기능보다 협업 구조를 먼저 고려하는 개발 습관 형성

### 8.2 아쉬운 점 및 개선 아이디어

- 초반 역할 분배가 더 명확했으면 충돌 더 줄일 수 있었음
- 향후 기능 연동성(데이터 공유 등) 고려해 설계하면 좋겠음
